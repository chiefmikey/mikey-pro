## Overview

Mikey Pro is a comprehensive ESLint 9 style guide published as `@mikey-pro/*` scoped npm packages. It ships 7 packages: a core ESLint flat config (50+ plugins), 4 framework-specific configs (React, Vue, Svelte, Angular), a Prettier config, and a Stylelint config. All packages share version 9.0.7.

## Architecture

### Config Composition Pattern

All configs are arrays of flat config objects (ESLint 9 pattern):

```
Core:      [ignores, baseConfig, ...baseOverrides]
React:     [ignores, baseConfig, reactConfig, ...baseOverrides]
Vue:       [ignores, baseConfig, vueConfig, ...baseOverrides]
Svelte:    [ignores, baseConfig, svelteConfig, ...baseOverrides]
Angular:   [ignores, baseConfig, angularConfig, angularTemplateConfig, ...baseOverrides]
```

Framework configs import `baseConfig` and `baseOverrides` from the core package via `file:../eslint-config` dependency links.

### Plugin Architecture

**Core (`base-config.js`)** loads 30+ plugins:

- Code quality: unicorn (~80 rules), sonarjs, promise, regexp, security, no-secrets
- Formatting: prettier, perfectionist, simple-import-sort, sort-destructure-keys
- File types: jsonc, yml, toml, html, markdown, shell
- Testing: jest, jest-dom, cypress, testing-library, no-only-tests
- Boundaries: boundaries, compat, filenames, css-modules, n
- Parsing: @babel/eslint-parser (JS), @typescript-eslint/parser (TS)

**Overrides (`overrides.js`)** provides 16 named configs:
`ts`, `css`, `scss`, `less`, `yaml`, `toml`, `md`, `packageJson`, `htmlConfig`, `jsonc`, `json5`, `jestJs`, `jestTs`, `cypressConfig`, `shellConfig`, `envFilesConfig`

### Key Design Decisions

- **`noInlineConfig: true`**: Consumers cannot override rules with inline ESLint comments
- **`reportUnusedDisableDirectives: true`**: Enforced globally
- **Zero-tolerance CI**: `--max-warnings 0` in code-quality workflow
- **Self-linting**: Root `eslint.config.js` imports directly from `./configs/eslint-config/index.js`

```bash

# Development

# Maintenance
npm run ncu                       # Check all packages for dependency updates
```

## Gotchas

1. **Two package.json files**: Root (`v0.0.0`, dev only) vs `configs/package.json` (`v9.0.7`, published meta-package). Easy to edit the wrong one.

2. **Sub-packages have independent node_modules**: Each config in `configs/` manages its own deps. `npm run npm:install` handles all of them, but forgetting this after cloning will cause failures.

3. **Self-linting creates circular behavior**: Changes to `configs/eslint-config/` immediately affect linting of the project itself. A broken rule will prevent linting the codebase.

4. **`noInlineConfig: true` is aggressive**: Consumers cannot use `// eslint-disable` comments. This is intentional but will surprise users who expect the standard behavior.

5. **`eslint-plugin-unicorn` version mismatch**: `^61.0.2` in `configs/eslint-config/package.json` vs `^62.0.0` in root `package.json`. Could cause resolution conflicts.

6. **`rules.js` and `base-config.js` appear to duplicate rule definitions**: 623 lines in `rules.js` mirror the rules in `base-config.js`. Unclear which is the source of truth for consumers importing `./rules`.

7. **Test timeout of 10s**: Loading 30+ plugins is slow. Tests that add more plugin initialization may need even longer timeouts.

8. **`eslint-plugin-shell` at v0.0.1**: Essentially a prototype. High risk for ESLint 10 breakage and may not handle edge cases well.

9. **Release workflow is complex**: 226 lines across 3 jobs with conditional logic. The last 10 commits were all release workflow fixes — this area is fragile.

10. **Stale local branch**: `feat/add-shell-env-support` was already merged via PR #28 but the local branch remains. Can be safely deleted.
