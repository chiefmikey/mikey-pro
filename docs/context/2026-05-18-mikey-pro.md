## Overview

Mikey Pro is a comprehensive AI guardrails suite published as `@mikey-pro/*` scoped npm packages. It ships ESLint 10 flat configs (core + 4 framework variants), Prettier, Stylelint, and Python Ruff configs. The unified base package `mikey-pro` re-exports ESLint flat config and is consumed by all framework variants. Core purpose: enforce consistent, AI-safe code quality patterns across web and Python projects.

## Architecture

### Config Composition Pattern

All ESLint configs are arrays of flat config objects (ESLint 10 pattern):

```
Core:      [ignores, baseConfig, ...baseOverrides]
React:     [ignores, baseConfig, reactConfig, ...baseOverrides]
Vue:       [ignores, baseConfig, vueConfig, ...baseOverrides]
Svelte:    [ignores, baseConfig, svelteConfig, ...baseOverrides]
Angular:   [ignores, baseConfig, angularConfig, angularTemplateConfig, ...baseOverrides]
```

Framework configs import `baseConfig` and `baseOverrides` from base config via `mikey-pro/eslint/*` subpath exports.

### Plugin Architecture

**Core (`base-config.js`)** loads 25+ plugins:
- Code quality: unicorn (~80 rules), sonarjs, promise, regexp, security, no-secrets
- Formatting: prettier, perfectionist, simple-import-sort, sort-destructure-keys
- File types: jsonc, yml, toml, html, markdown, shell
- Testing: jest, jest-dom, cypress, testing-library, no-only-tests
- Boundaries: boundaries, compat, filenames, css-modules, n
- Parsing: @babel/eslint-parser (JS), @typescript-eslint/parser (TS)

**Framework plugins:**
- React: jsx-a11y, react, react-hooks, react-perf, react-refresh
- Vue: vue, vue-eslint-parser
- Svelte: svelte, svelte-eslint-parser
- Angular: @angular-eslint/template, @angular-eslint/eslint-plugin

### Key Design Decisions

- **Peer dependencies pattern**: eslint, prettier, stylelint are peerDeps (not regular deps) to avoid npm hoisting conflicts when plugins have older peer ranges
- **Unified base**: `mikey-pro` re-exports ESLint flat config; framework configs depend on `mikey-pro` transitively and import from its subpaths
- **All packages share a single version**: bumped together via `scripts/bump-version.js`
- **`noInlineConfig: false`**: allows targeted `// eslint-disable` for edge cases
- **Self-linting**: Root `eslint.config.js` imports directly from `./configs/eslint-config/index.js`

## Hardcoded Versions to Watch

These require manual updates when dependency versions change:

- **React version** in `eslint-config-react/index.js`: `settings.react.version: '19'` — bump when React 20 releases. Cannot use `'detect'` due to ESLint 10 removal of `getFilename()`.
- **Python target** in `ruff-config/ruff.toml`: `target-version = "py312"` — update when targeting newer Python
- **Node.js version** in `base-config.js` `n/no-unsupported-features/es-syntax`: `version: '>=20.0.0'` — update when minimum Node version changes

## Known Gotchas

1. **Version drift resurfaces if not bumped atomically**: `eslint-config-react` (10.3.2) and `prettier-config` (10.2.0) are drifted. Must use `npm run release:bump X.Y.Z` to sync all 9 files at once.

2. **Sub-packages have independent node_modules**: Each config in `configs/` manages its own deps. Must run `npm run npm:install` after cloning. Not captured in any lockfile.

3. **Publishing order matters**: `mikey-pro` (unified base) must publish first since framework configs depend on it transitively. Use `npm run publish:all` or publish in order.

4. **Self-linting creates circular dependency**: Changes to `configs/eslint-config/` immediately affect project linting. A broken rule will prevent linting the codebase.

5. **`noInlineConfig: false` is permissive**: Consumers can override rules with inline comments, which may or may not be intentional.

6. **Prettier drift on main**: 21 files have formatting mismatches as of this session. CI gate at `--max-warnings 0` is blocking all PRs.

7. **Test regression**: Test duration 10x slower (445s vs 45s) and 33 of 209 tests failing. Root cause unknown; being investigated by parallel worker.

8. **No v10 release tags on GitHub**: All v10.x versions were published to npm but never tagged on GitHub. Next release should create v10.3.3 tag.

9. **theme-adobe/ not in Prettier ignore**: Current `.prettierignore` excludes `theme/` but not `theme-adobe/`. Future cleanup needed.

10. **Two theme dirs remain**: `theme/` (v0.0.1, part of suite) and `theme-adobe/` (separate variant). One remains unignored by Prettier.
